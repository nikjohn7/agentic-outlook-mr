How to read AI engine outputs

A plain-language guide to the files each run produces.

Each run reads a set of fund and asset-manager outlook documents and produces two spreadsheets and one Word document. output.csv lists every allocation call we kept, one row per call. failures-client.csv lists everything we considered but did not keep, and why. The one-page firm summaries Word document gives a readable, per-firm digest of each manager's views. This guide explains the two spreadsheets and the actions to take.

output.csv — the calls we kept

Each row is one allocation call: a view on one sub-asset class, taken from one source document, with the supporting text. The first ten columns are the core output; the last six help you review it.

ColumnWhat it means
FirmThe asset manager that published the document.
DateThe document's own publication date, as DD/MM/YYYY. It is read from the document itself, not from any source list; it is left blank when the document does not state a date.
SourceThe document title. When two related documents from the same firm are read together, their titles are joined with a |.
URLA link to the document (also joined with a | for grouped documents).
Sub-Asset ClassThe specific asset the call is about (e.g. "Emerging Market Equities"), taken from the approved asset list.
Asset Class CategoryThe broad grouping the sub-asset sits under. Filled in automatically from the asset list.
Canva GroupingsThe display grouping for the sub-asset. Filled in automatically from the asset list.
Asset ClassThe mid-level asset class for the sub-asset. Filled in automatically from the asset list.
ViewThe call: O overweight, N neutral, U underweight, UNCERTAIN when the document does not take a clear position.
Full CommentaryThe reasoning plus the exact supporting text (and its page or location), so you can see why the call was made and check it. When one call is supported by more than one document (or more than one place in a document), the commentary shows each source's wording as a separate labeled segment — Source Title (locator): … — joined by  |||| , so no supporting evidence is lost when calls are merged.
confidenceA 0–100 score from fixed rules (evidence quality, how clearly the view is stated, whether a second reader confirmed it). It is not a model's own opinion of its confidence.
bandThe confidence score in words: High, Medium, or Low.
review_flagWhether a human should take a closer look. review means yes; none means it passed cleanly.
basisHow the call was reached: stated (the document says it directly), inferred (we read it as a reasonable consequence of what the document says), or forecast_delta (derived from a change in a forecast).
checker_strengthHow strongly a second reader confirmed the evidence: decisive, adequate, or thin.
call_languageHow the view was expressed in the document — e.g. an explicit dial/grid, an explicit stance in prose, a directional lean, or an implied reading.

failures-client.csv — what we considered but did not keep

This file records candidate calls that were set aside, and why. Most entries are healthy: duplicates that were kept once, near-duplicate calls, or views whose supporting text could not be verified and so were dropped rather than kept unconfirmed. A large failures file is normal and usually means the filtering is working. Rows are grouped by their What happened label and sorted with the most important labels first: entries that may need a decision or a re-run are at the top, and no-action housekeeping entries (duplicates, too-small changes) are at the bottom. The table below lists the labels in that same order:

What happenedMeaning
Document could not be ingestedA whole source couldn't be fetched or converted into text, so it was skipped while the rest of the run continued. Let us know so we can re-run it.
Conflicting views — none keptTwo views disagreed and could not be reconciled automatically; neither was kept. A human should decide.
Part of a document couldn't be read or processed / Second-reader check couldn't run / Skipped — source details missingA processing issue affected part of a document. Let us know so we can re-run it.
Suggestion — review recommendedAn implied reading challenges the firm's stated call; the stated call was kept and the challenge recorded for review.
Conflicting views — other call keptTwo documents disagreed; the more current/specific call was kept. Review if the kept call looks wrong.
Evidence could not be verified (or "… verified visually")The supporting text couldn't be confirmed — even after a visual page check, for the "visually" variant — so the call was dropped rather than kept unverified. A human can check the cited page.
Evidence location missingA chart/table call didn't record where it came from, so it couldn't be verified.
Evidence points the other way / Evidence is about a different asset / Not a forward-looking viewA second reader found the cited text doesn't back the call as made, so it was dropped. A human can check the cited page.
Skipped — asset not on the listThe asset named didn't match the approved list, so it was left out.
Change too small to callThe forecast change behind the call was too small to justify a view.
Superseded — a newer document from this firm updated this viewAnother document from the same firm took a different view on this asset; a more current (or otherwise preferred) call was kept in its place. Review if the kept call looks wrong.
Merged into a related call — please reviewThe firm made views on two closely related assets (e.g. a broad category and a specific part of it) that pointed different ways; they were read as one collective call and the most relevant view was kept. Review that the kept view is the right one.
Duplicate — already coveredThe same view for this asset appears in another of the firm's documents; kept once, with the other document's wording merged into the kept row's commentary. No action needed.
Duplicate — same call, related assetRepeats a view already kept for a closely related asset from the same text; kept once. No action needed.
Merged — same view stated in several documentsSeveral of the firm's documents made the same call on this asset; they were merged into one row and every document's wording is preserved in that row's commentary. No action needed.
Merged — same view on a closely related assetThe firm made the same call on two closely related assets (e.g. a broad category and a specific part of it); they were merged into one row on the most precise asset and every document's wording is preserved. No action needed.

The Explanation and Evidence / notes columns in the file give the one-line meaning and the underlying text for each row. Because the file is sorted most-important-first, start at the top: act when you see "Document could not be ingested", "Conflicting views — none kept", "Suggestion — review recommended", or a repeated "Part of a document couldn't be…" for a document you expected calls from.

What to do with the output

  • Review the flagged rows first. In output.csv, sort or filter on review_flag = review — these are the calls we think most warrant a human eye.
  • Treat UNCERTAIN as a deliberate abstention, not an error. It means the document did not take a clear position on that asset; we chose not to invent one.
  • Spot-check a call. Open the Full Commentary, read the quoted evidence, and confirm it against the cited page or location in the source document.
  • Send back anything that looks wrong. If a kept call reads the wrong way, an expected call is missing, or a whole document produced no calls, let us know the Firm, Source, and Sub-Asset Class so we can look into it.
