Problem Statement

Markets Recon receives fund and asset manager outlook documents in PDF or HTML-derived formats. The POC should identify the asset and sub-asset classes referenced in each source, determine whether each relevant sub-asset is overweight, neutral, or underweight, and retain enough evidence for an analyst to verify the result quickly.

Recommended Next Action

Build a local-first POC pipeline that proves the full extraction and review workflow before investing in a production front end. The first milestone should focus on reliable ingestion, Allocator Pro taxonomy alignment, analyst-calibrated call extraction, deterministic confidence scoring, and a simple reviewable output.

POC Workflow

01
Input folder Place one or more source files into a designated folder and start the process.
02
Auto-detect source Classify input as PDF or HTML/scraped output and use the relevant ingestion path.
03
Normalize content Extract text, tables, sections, page/source references, and searchable chunks.
04
Detect topics + taxonomy Extract topics, asset classes, and corresponding Allocator Pro sub-asset classes.
05
Make + verify calls Produce OW/N/UW calls, score confidence, and route uncertain items to review.

Analyst Brain

The models should not rely on generic finance prompts alone. They need a compact analyst framework: examples of source excerpts, approved calls, rejected interpretations, and the reasoning a Markets Recon analyst would apply.

Confidence Rubric

Confidence should be computed from a documented rubric covering direction evidence, citation support, taxonomy certainty, rationale support, ambiguity, and source quality. Low confidence items should be flagged instead of silently accepted.

Call Output Shape

CSV exportSimple downstream format for Allocator Pro ingestion or Excel review.
Run outputOne file per run, referencing each processed source and its calls.
Evidence trailEach accepted call keeps page/source references and rationale.

Run-Level Output

Each processing run should create a single reviewable output file containing links or references to all files processed in that run, with the detected asset classes, sub-asset classes, calls, confidence bands, and review flags shown per source file. To preserve readability and output size, each run should contain no more than 20 input items.

In Scope

  • Folder-based PDF and HTML/scraped-file input.
  • Asset class, sub-asset, topic, and keyword extraction.
  • OW/N/UW call generation with citations.
  • Confidence scoring and review routing.
  • CSV and simple HTML review output.

Out of Scope for POC

  • Markets Recon production front-end build.
  • Full website integration and user management.
  • Large-scale batch infrastructure.
  • OCR/vision for heavily scanned documents, unless separately agreed.

Assumptions

  • Markets Recon provides scraped HTML where website scraping is required.
  • The POC can run locally or on a designated Markets Recon machine.
  • Public source documents can use cost-efficient model stages.
  • Allocator Pro taxonomy labels are the target output labels.

Model Strategy

Use lower-cost models for extraction, document cleaning, topic detection, and initial taxonomy matching. Use stronger models for call-making, reasoning, and verification. A practical setup may combine OpenAI and Claude models, with optional cheaper API routes such as OpenRouter or DeepSeek for low-risk public-data extraction if Markets Recon is comfortable with that.

Verification Strategy

Use an LLM-as-judge pattern for higher-risk calls: multiple model passes or model instances independently assess the evidence and then reconcile disagreements. The final confidence score should still be rubric-based, not a free-form model confidence estimate.

Key Decisions Needed from Markets Recon

Preferred POC runtime Confirm whether the first version should run locally, on a designated internal machine, or hosted lightly.
Output preference Confirm whether CSV alone is enough for testing, or whether the HTML review file should be the primary analyst output.

What I Would Need

  • A representative set of source PDFs and scraped HTML outputs.
  • The latest Allocator Pro taxonomy or confirmation that the current website taxonomy is canonical.
  • Alex's spreadsheet and guidance on how reliable each existing call should be treated.
  • Analyst-reviewed examples to build the call-making framework and confidence rubric.
  • API/provider budget guidance and any compliance restrictions around model vendors.
  • Agreement that the first POC can be local-first and front-end independent.

A successful POC should process representative documents, produce Allocator Pro-compatible call outputs, expose source evidence, separate high-confidence calls from review items, and demonstrate whether the system can match or improve on the benchmark spreadsheet. Indicative timeline: three weeks end to end.
